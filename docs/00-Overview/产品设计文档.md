# 林间空地 Lichtung - 产品设计文档

> **产品名称**：林间空地 Lichtung
> **命名寓意**：海德格尔哲学中的"澄明之境"——在算法密林中显现真实的空地
> **项目定位**：研究用Demo，验证"多数错觉可视化"干预对用户感知极化的校准效果
> **受众**：下一阶段的实施团队（AI或人类开发者）
> **时间估算**：3-4天完成MVP
> **文档版本**：v2.0
> **最后更新**：2026-02-23
> **变更说明**：v2.0 重构了核心交互模型——从"悬停Tooltip"改为"评论区默认曝光 + 独立数据按钮"双通道方案；数据结构改为支持 4–7 级可变量表

* * *

## 1. 背景与研究动机

### 1.1 核心问题

**多数错觉（Majority Illusion）**：

* Lerman et al. (2016)在*PLOS ONE*发现：社交网络的异配结构中，仅20%的节点活跃即可导致60-70%的节点产生"该观点为多数"的错误感知
* 算法推荐系统放大这一效应：高参与度内容（点赞、转发、评论）被过度呈现，用户误以为"我看到的=真实分布"

**现实影响**：

* 用户基于偏差样本形成对"外群体立场"的误判
* 夸大对方敌意程度（Moore-Berg et al., 2020）
* 形成"大家都很激进"的错觉

### 1.2 干预假设

**如果**在用户即将接触评论区极端观点之前，先展示真实的公众态度分布数据，**那么**用户对意见分布的估计将更准确，从而降低感知极化——具体表现为修正用户对"对方阵营"极端程度的过度估计。

**注意**：干预目标不是改变用户立场，也不是利用多数压力施压少数派，而是修正用户对对立群体内部异质性的感知。

### 1.3 设计目标

**不是**：

* ❌ 改变用户的政治立场
* ❌ 推荐"更好"的内容
* ❌ 做完整的新闻产品
* ❌ 用多数人的立场施压少数派（从众压力工具）

**而是**：

* ✅ 让用户意识到"我看到的评论 ≠ 真实的公众态度分布"
* ✅ 修正用户对对方阵营极端程度的过度估计
* ✅ 校准用户对意见分布的感知
* ✅ 验证"分布可视化"的干预效果

## 2. 产品设计方案

### 2.1 核心功能

**Feed流 + 评论区默认曝光 + 独立数据按钮**

**设计理念**：

* 默认曝光（Default Exposure）：不阻断用户操作，但将权威分布数据放置在用户必经的视觉路径上
* 体验式学习（Experiential Learning）：让用户自己发现评论区极端样本与真实分布的差异
* 认知失调（Cognitive Dissonance）：通过直方图与评论区的并置，触发"评论区的声音 vs 真实分布"的冲击

**为什么干预点是评论区而非报道本身**：

* 看报道时，用户接收的是事实信息，尚未形成对"对方阵营"的感知
* 评论区才是感知极化的制造现场——用户在此看到对方阵营最激进、最高赞的发言，并将极端样本当作对方的全貌
* 精确干预时机：用户已产生"想看看别人怎么想"的意图，但尚未被极端评论锚定认知的窗口

### 2.2 用户旅程

```
用户打开网站
    ↓
滑动Feed流，浏览多个事件的报道
    ↓
看到一篇关于堕胎议题的报道，卡片上有轻量议题标签角标
    ↓
[路径A：被动干预 — 主干路径]
点击「💬 评论」按钮，想看看别人怎么想
    ↓
评论区展开，顶部固定显示 Pew 调查直方图
（不可折叠、不可关闭，自然占据评论区第一屏）
    ↓
用户看到：大部分人态度温和，极端立场是少数
    ↓
向下滚动，看到评论区的极端发言
    ↓
[认知冲击] "评论区吵得这么凶，但刚才数据显示大部分人其实没那么极端"
    ↓
[自我发现] "评论区的声音不代表真实的民意分布"

---

[路径B：主动探索 — 辅助路径]
用户对某议题好奇，点击卡片上的「📊 公众态度数据」按钮
    ↓
仅展开直方图，不显示评论
    ↓
用户主动查看该议题的公众态度分布
```

**直方图组件规格**：

* 位置：评论区最顶部，固定不可折叠、不可关闭
* 宽度：与卡片等宽
* 背景：`#F8F9FA`（极浅灰，与评论区白色区分）
* 边框：1px solid `#E8E8E8`
* 圆角：8px
* 内边距：16px
* 底部间距：与评论列表间距 12px

**直方图视觉规范**：

* 图表类型：水平分布直方图（Horizontal Bar Chart）
* X 轴：受访者百分比
* Y 轴：4–7 级态度刻度（档数和标签由原始数据源决定）
* 柱状条高度：24px，间距 8px
* 柱状条圆角：右侧 4px
* 百分比标签：柱状条右侧，13px，`#536471`
* 视觉重点：以面积（柱子长度）对比自然凸显"温和派/中间档占多数"的事实

**色彩方案（暖色系非政治中性色，参考 Pew Research 图表风格）**：

* 设计原则：使用暖色调中性色，各档位可区分但不暗示阵营对立或政治属性
* 严禁使用红/蓝（美国政党色）、红/绿（对立暗示）等带有政治属性的配色
* 推荐色板（以 4 档为例）：
  * 档位 1：`#D4A574`（沙棕，Sand）
  * 档位 2：`#C4956A`（焦糖，Caramel）
  * 档位 3：`#9C8AA5`（灰紫，Mauve）
  * 档位 4：`#7B6B8A`（深灰紫，Plum Grey）
* 档数可变时（5–7 档）：在上述色板基础上等距插值，保持暖色调中性风格
* 配色逻辑：量表两端分属不同色相（暖棕 → 灰紫），中间档位为过渡色，避免任何一端的颜色被解读为"好"或"坏"

**桥接文案（必须）**：

直方图前必须配有说明文案，消除与报道之间的突兀感。

格式：
> 「根据 [数据来源] [年份] 的调查，美国公众对[议题名称]的态度分布如下：」

### 3.4 独立数据展开态（辅助探索路径）

用户点击「📊 公众态度」按钮后，仅在卡片下方展开直方图组件，不显示评论。

复用与 3.3 相同的直方图组件，区别仅在于不附带评论列表。

### 3.5 Feed 卡片议题标签角标

报道卡片上保留轻量视觉指示（议题标签角标，如「堕胎权利」「枪支管控」），告知用户此报道已匹配到预设议题。

* 仅在报道匹配议题时显示
* 匹配失败时不显示角标和「📊 公众态度」按钮

## 4. 技术架构

### 4.1 技术栈选择

**推荐：Vite + React + TypeScript**

**理由**：

* ✅ Vite：极快的开发服务器
* ✅ React：组件化，易于状态管理
* ✅ TypeScript：类型安全，减少bug
* ✅ 轻量：不需要Angular的复杂性

**替代方案**：

* Vite + Vue 3（如果更熟悉Vue）
* 纯HTML + Vanilla JS（如果极简化）

## 5. 实施步骤

### Phase 1: 数据准备（半天）

**任务**：

1. 确认 Pew 原始调查报告中各档百分比数据（手动验证，不依赖 AI 生成）
2. 将已验证的民调数据结构化录入 `polling.json`
3. 生成模拟报道和模拟评论数据

**模拟数据生成策略**：

* 报道：基于真实新闻标题改写，每个议题 10-20 篇
* 评论：故意偏向极端，高赞评论以激进立场为主，模拟真实评论区的选择偏差
* 民调数据：严格使用 Pew/ANES 原始数据，不编造

## 6. 关键代码示例

### 6.1 FeedItem 组件

```tsx
import { useState } from 'react';
import { Report, Topic } from '../types';
import CommentSection from './CommentSection';
import DistributionChart from './DistributionChart';
import TopicBadge from './TopicBadge';

interface Props {
  report: Report;
  topic?: Topic;        // 匹配到的议题（可能无）
}

export default function FeedItem({ report, topic }: Props) {
  const [showComments, setShowComments] = useState(false);
  const [showDataOnly, setShowDataOnly] = useState(false);

  const hasTopic = !!topic;

  return (
    <div className="feed-item">
      {/* 议题标签角标 */}
      {hasTopic && <TopicBadge topic={topic} />}

      {/* 报道内容（点击跳转外部新闻源） */}
      <a href={report.sourceUrl} target="_blank" rel="noopener noreferrer">
        <h3>{report.title}</h3>
        <p>{report.summary}</p>
      </a>

      <div className="meta">
        🔗 来源：{report.source} • {formatTime(report.publishedAt)}
      </div>

      {/* 操作按钮 */}
      <div className="actions">
        <button onClick={() => {
          setShowComments(!showComments);
          setShowDataOnly(false);
        }}>
          💬 评论
        </button>

        {hasTopic && (
          <button onClick={() => {
            setShowDataOnly(!showDataOnly);
            setShowComments(false);
          }}>
            📊 公众态度
          </button>
        )}
      </div>

      {/* 评论区展开（含顶部直方图） */}
      {showComments && (
        <CommentSection topicId={report.topicId} />
      )}

      {/* 独立数据展开（仅直方图） */}
      {showDataOnly && (
        <DistributionChart topicId={report.topicId} />
      )}
    </div>
  );
}
```

### 6.2 DistributionChart 组件（N 档可变直方图）

```tsx
import { PollingData } from '../types';

interface Props {
  pollingData: PollingData;
}

// 生成暖色系中性色梯度：从沙棕到灰紫，档数可变
function getBarColors(count: number): string[] {
  // 色板锚点：暖棕端 → 灰紫端
  const warmEnd  = [212, 165, 116]; // #D4A574 Sand
  const coolEnd  = [123, 107, 138]; // #7B6B8A Plum Grey
  return Array.from({ length: count }, (_, i) => {
    const ratio = count === 1 ? 0.5 : i / (count - 1);
    const r = Math.round(warmEnd[0] + (coolEnd[0] - warmEnd[0]) * ratio);
    const g = Math.round(warmEnd[1] + (coolEnd[1] - warmEnd[1]) * ratio);
    const b = Math.round(warmEnd[2] + (coolEnd[2] - warmEnd[2]) * ratio);
    return `rgb(${r}, ${g}, ${b})`;
  });
}

export default function DistributionChart({ pollingData }: Props) {
  const maxPercent = Math.max(...pollingData.distribution);
  const colors = getBarColors(pollingData.scaleLabels.length);

  return (
    <div className="distribution-chart">
      {/* 桥接文案 */}
      <p className="bridging-text">{pollingData.bridgingText}</p>

      {/* N 档直方图 */}
      <div className="bars">
        {pollingData.scaleLabels.map((label, index) => (
          <div key={index} className="bar-row">
            <span className="bar-label">{label}</span>
            <div className="bar-track">
              <div
                className="bar-fill"
                style={{
                  width: `${(pollingData.distribution[index] / maxPercent) * 100}%`,
                  backgroundColor: colors[index],
                }}
              />
            </div>
            <span className="bar-percent">
              {pollingData.distribution[index]}%
            </span>
          </div>
        ))}
      </div>

      {/* 数据来源标注 */}
      <p className="source-note">
        数据来源：{pollingData.source} ({pollingData.surveyYear})
      </p>
    </div>
  );
}
```

## 7. 设计原则

### 7.1 默认曝光（Default Exposure）

**核心**：不阻断用户操作，不要求额外动作，但将权威数据放在用户必经的视觉路径上。研究表明即使用户仅快速扫视，锚定效应仍会在后续阅读评论时起作用。

**不做**：

* ❌ 弹窗警告"你的看法可能有偏差"
* ❌ 将直方图作为查看评论的强制门槛（会扼杀评论参与度）
* ❌ 默认折叠直方图（等同于自愿入口，曝光率过低）
* ❌ 阻断用户的阅读流程

**而做**：

* ✅ 评论区顶部固定展示直方图，不可折叠、不可关闭
* ✅ 用户向下滚动即可查看评论，无任何阻断
* ✅ 另设独立数据按钮供主动探索
* ✅ 保持阅读的流畅性

### 7.2 体验式学习

**核心**：让用户自己发现评论区极端样本与真实分布的矛盾，而非被动告知

**不加**：

* ❌ "评论区夸大了反对声音"
* ❌ "你的判断可能有误"
* ❌ 任何带有说教意味的文案

**依赖**：

* ✅ 直方图与极端评论的并置产生的自然对比
* ✅ 用户的自我意识："评论区吵得这么凶，但数据显示大部分人其实没那么极端？"
* ✅ 认知失调的自然触发

### 7.3 学术严谨性

**数据真实性**：

* 仅使用预先验证的外部权威民调数据（Pew Research / ANES）
* 直方图档数由原始数据源决定（4–7 级），不人为增减
* 严禁使用平台内部数据、用户投票等生成图表
* 严禁使用"支持 vs 反对"二元数据

**单一变量**：

* 不叠加其他干预（如情绪过滤、去标签）
* 保证测量的是"分布可视化"的效果

**可测量**：

* 记录评论按钮点击率 vs 独立数据按钮点击率
* 记录用户在直方图区域的停留时间
* 前后测：用户对分布的估计准确度
* A/B对照：有/无直方图的评论区对比

## 10. 风险与应对

### 风险1：用户快速跳过直方图

**表现**：用户点开评论区后直接滚到评论，不看直方图

**应对**：

* 默认曝光设计本身已确保视觉接触——即使快速扫视，锚定效应仍然存在
* 可通过 A/B 测试量化"扫视 vs 不展示"两种情况下的感知极化差异
* 首次访问时可加轻量引导提示

### 风险2：直方图太复杂

**表现**：用户看不懂直方图

**应对**：

* 桥接文案用自然语言解释量表含义
* 采用水平柱状图（最直觉的图表形式）
* 内测时收集反馈，必要时简化标签措辞

### 风险3：数据质量

**表现**：Pew 原始数据的量表设计不统一（4档/5档/7档混存）

**应对**：

* 直方图组件设计为 N 档可变，不硬编码档数
* 每个议题独立维护量表标签和分布数据
* Demo 阶段仅选 2–3 个已确认数据可用的议题

### 风险4：用户认为平台有立场偏见

**表现**：用户觉得展示"大部分人态度温和"是平台在暗示极端立场是错的

**应对**：

* 直方图仅呈现数据，不附加任何价值判断
* 桥接文案保持中性描述（"态度分布如下"），不使用"实际上""真相是"等引导性措辞
* 明确标注数据来源和调查年份，增强可信度
